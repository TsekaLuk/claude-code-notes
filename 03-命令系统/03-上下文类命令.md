# 上下文类命令 — Claude Code 源码分析

> 模块路径：`src/commands/context/`、`src/commands/add-dir/`、`src/commands/copy/`、`src/commands/ctx_viz/`
> 核心职责：展示、管理和操作当前对话的上下文窗口状态与工作目录配置
> 源码版本：v2.1.88

## 一、模块概述

上下文类命令帮助用户理解和管理 Claude 的"感知范围"。`/context` 以可视化网格展示当前 token 使用情况；`/add-dir` 将额外目录加入 Claude 的工作空间权限；`/copy` 将 Claude 最后一条回复复制到剪贴板；`ctx_viz` 是内部调试工具（列在 `INTERNAL_ONLY_COMMANDS`），用于上下文可视化分析。

这类命令的共同特点是：**直接反映并操作 `ToolUseContext` 和 `AppState`**，让用户能看到和调整 AI 可以访问的信息范围。

## 二、架构设计

### 2.1 核心类/接口/函数

**`context`（`src/commands/context/index.ts`）**
实际上由两个命令组成：`context`（交互模式，`LocalJSXCommand`，渲染彩色网格）和 `contextNonInteractive`（非交互模式，`LocalCommand`，输出纯文本）。通过 `getIsNonInteractiveSession()` 判断当前模式，确保两者只有一个处于启用状态（`isEnabled` + `isHidden`）。

**`analyzeContextUsage`（`src/utils/analyzeContext.ts`）**
`/context` 命令的核心数据收集函数，被交互模式（`context.tsx`）和非交互模式（`context-noninteractive.ts`）共用。先应用与 `query.ts`（实际 API 调用）相同的消息转换（compact boundary、projectView、microcompact），确保显示的 token 数与模型实际接收的一致。

**`collectContextData`（`src/commands/context/context-noninteractive.ts`）**
非交互路径共用的数据收集层，供 SDK `get_context_usage` 控制请求和 `/context` 命令复用，避免重复实现转换逻辑。

**`addDir`（`src/commands/add-dir/index.ts`）**
类型：`LocalJSXCommand`，`argumentHint: '<path>'`。核心逻辑在 `add-dir.tsx` 中，通过 `applyPermissionUpdate` + `persistPermissionUpdate` 将新目录加入会话级或持久化的权限上下文。支持带路径直接执行和交互式选择两种模式。

**`copy`（`src/commands/copy/index.ts`）**
类型：`LocalJSXCommand`。功能：复制 Claude 最后一条（或第 N 条）回复到剪贴板。`argumentHint` 说明支持 `/copy N` 格式指定历史回复编号。

## 三、核心实现走读

### 3.1 关键流程

**`/context` 精确 token 计算的关键：**
1. 获取 `messages` 后先调用 `toApiView()` 进行与实际 API 调用相同的预处理
2. 关键在于 `CONTEXT_COLLAPSE` 功能：若启用，`projectView()` 会折叠重复的文件内容视图（如多次 Read 同一文件），这与 API 调用前完全相同的处理
3. 再经过 `microcompactMessages` 处理（剔除冗余 token）
4. 最终分析的 `compactedMessages` 与实际发送给模型的消息高度一致，确保显示的 token 数"所见即所得"

**`/add-dir` 双路径设计：**
1. 若用户提供 `args`（目录路径），直接调用 `handleAddDirectory(path)`
2. 若无 `args`，渲染交互式目录选择 UI（`AddWorkspaceDirectory` 组件）
3. `handleAddDirectory` 构建 `permissionUpdate`，决定持久化级别：`remember=true` 写入 `localSettings`，否则仅 `session` 级别
4. 调用 `applyPermissionUpdate` 更新内存中的 `toolPermissionContext`
5. 调用 `persistPermissionUpdate` 持久化到配置文件

### 3.2 重要源码片段

**`/context` API 视图转换（`src/commands/context/context.tsx`）**
```typescript
// 必须镜像 query.ts 的处理，否则显示值与实际不符
function toApiView(messages: Message[]): Message[] {
  let view = getMessagesAfterCompactBoundary(messages)
  if (feature('CONTEXT_COLLAPSE')) {
    const { projectView } = require('../../services/contextCollapse/operations.js')
    view = projectView(view) // 折叠重复内容，减少过计算
  }
  return view
}
```

**`/context` 完整分析管线（`src/commands/context/context.tsx`）**
```typescript
// 三步：边界裁剪 → 微压缩 → 分析 → 渲染
const apiView = toApiView(messages)
const { messages: compactedMessages } = await microcompactMessages(apiView)
const data = await analyzeContextUsage(
  compactedMessages, mainLoopModel,
  async () => appState.toolPermissionContext, tools,
  appState.agentDefinitions, terminalWidth, context,
  undefined, apiView  // 传入原始 apiView 供 API usage 提取
)
const output = await renderToAnsiString(<ContextVisualization data={data} />)
```

**`/add-dir` 权限更新（`src/commands/add-dir/add-dir.tsx`）**
```typescript
// 区分会话级（session）和持久化（localSettings）权限
const permissionUpdate = {
  type: 'addDirectories' as const,
  directories: [path],
  destination: remember ? 'localSettings' : 'session',
}
const updatedContext = applyPermissionUpdate(
  latestAppState.toolPermissionContext, permissionUpdate
)
```

**`/context` 双模式注册（`src/commands/context/index.ts`）**
```typescript
// 交互模式：彩色网格，仅在交互会话启用
export const context: Command = {
  type: 'local-jsx',
  isEnabled: () => !getIsNonInteractiveSession(),
  load: () => import('./context.js'),
}
// 非交互模式：纯文本，仅在非交互会话启用（如 SDK 调用）
export const contextNonInteractive: Command = {
  type: 'local',
  supportsNonInteractive: true,
  isEnabled() { return getIsNonInteractiveSession() },
  get isHidden() { return !getIsNonInteractiveSession() },
  load: () => import('./context-noninteractive.js'),
}
```

### 3.3 设计模式分析

**双模式命令（Dual-Mode Command）**：`/context` 是同一名称、两种实现的典型案例——运行时检测 `getIsNonInteractiveSession()` 决定激活哪个命令对象。两者在 `commands.ts` 中都被注册（`context` 和 `contextNonInteractive`），但互斥的 `isEnabled` 确保任意时刻只有一个生效。这比单一命令内部 if-else 分支更清晰，且两种模式的 `load()` 函数可独立懒加载。

**数据层与展示层分离**：`analyzeContextUsage` 和 `collectContextData` 负责数据收集和计算，`ContextVisualization` 组件负责渲染，`renderToAnsiString` 将 React 组件转为 ANSI 字符串（非交互模式下直接输出）。展示层对数据层无感知，数据层对终端环境无感知。

**权限更新的不可变模式（Immutable Update）**：`applyPermissionUpdate` 返回新的 `toolPermissionContext`（而非修改原有对象），遵循不可变更新原则，符合项目的 immutability 规范。

## 四、高频面试 Q&A

### 设计决策题

**Q1：`/context` 为什么必须模拟 `query.ts` 的消息处理步骤，而不是直接统计 `context.messages.length`？**

A：Claude Code 的 REPL 层（UI 层）保留的 `messages` 包含 snip 标记前的历史（用于 UI 滚动回放），以及 `CONTEXT_COLLAPSE` 功能折叠前的重复文件读取记录。如果直接统计 `context.messages`，计算结果会虚高——用户会看到"上下文已使用 180k token"，但模型实际只接收了 120k token（因为折叠了重复内容）。为了让 `/context` 显示"模型实际看到的数据"，必须执行相同的转换：`getMessagesAfterCompactBoundary` + `projectView` + `microcompact`。这是一致性原则的实践。

**Q2：`/add-dir` 中 `session` 和 `localSettings` 两种持久化级别如何选择？**

A：`session` 级别仅在当前会话有效，Claude Code 重启后恢复到原始配置，适合临时需要访问某个目录的场景。`localSettings` 持久化到项目的本地配置文件（非全局），下次在同一项目目录启动 Claude Code 时仍然生效，适合项目级别的常规工作目录。不选择写入 `globalSettings` 是因为工作目录通常与项目绑定，全局持久化会污染其他项目。`remember=true` 参数（UI 中的"Remember for this project"选项）控制这个选择。

### 原理分析题

**Q3：`ContextVisualization` 使用 `renderToAnsiString` 的原因是什么？**

A：`LocalJSXCommand` 的执行结果通常是返回 `ReactNode`，由 Ink 运行时渲染到终端。但 `/context` 的实现选择了 `renderToAnsiString`（将 React 组件渲染为带 ANSI 颜色码的字符串）并通过 `onDone` 输出，而不是直接返回 `ReactNode`。原因是 `/context` 需要同时支持交互模式和非交互模式——非交互模式下没有 Ink 运行时，无法渲染 `ReactNode`；使用 ANSI 字符串可以统一输出格式。这也使得 `/context` 的输出可以被管道捕获（`claude /context | grep "system prompt"`）。

**Q4：`validateDirectoryForWorkspace` 在 `add-dir` 中做哪些验证？**

A：基于文件系统安全性考虑，通常包含：路径存在性检查（目录必须存在）、路径解析（展开 `~`、相对路径到绝对路径）、权限检查（用户对该目录是否有读权限）、沙箱合规性检查（若启用沙箱模式，路径需在允许范围内）。`addDirHelpMessage` 在验证失败时提供友好的错误提示和使用示例，帮助用户理解正确的使用方式。

**Q5：`ctx_viz` 与 `/context` 的区别是什么？**

A：`ctx_viz` 是内部专用（`INTERNAL_ONLY_COMMANDS`）的调试命令，通过 JavaScript 索引文件（`index.js`，源码不完整）实现，主要用于 Anthropic 内部工程师调试上下文折叠和 token 计算逻辑，输出可能包含更详细的内部数据结构。`/context` 是面向用户的稳定命令，提供用户友好的可视化界面（彩色网格 + 百分比）。两者都基于 `analyzeContextUsage` 的核心计算，但展示层不同。

### 权衡与优化题

**Q6：`/context` 在每次调用时都执行 `microcompactMessages`，这不会影响性能吗？**

A：`microcompactMessages` 是规则式的 token 剔除（非 LLM 调用），速度较快，通常在毫秒级完成。`/context` 是诊断命令，用户主动调用，可以接受轻微延迟。关键是：**不执行 `microcompact` 会导致显示值与实际不符**——因为 API 调用前一定会执行 `microcompact`，跳过这步会导致显示虚高。宁愿稍慢但准确，也不快但误导用户。这是"数据准确性优先于响应速度"的权衡。

**Q7：为什么 `copy` 命令设计为支持 `/copy N` 格式？**

A：对话历史中 Claude 的多条回复有时语义上紧密相关，用户可能需要复制较早的某条完整回复（而非只复制最后一条）。`/copy N` 允许按"第 N 最新"的顺序索引，比绝对消息 ID 更直观。例如 `/copy 2` 复制倒数第二条 Claude 回复。这对于长时间开发会话中需要提取某个中间步骤的结果特别有用。

### 实战应用题

**Q8：如何在 CI/非交互环境中使用 `/context` 获取 token 使用情况？**

A：非交互模式下（`--print` 或环境检测为非 TTY），`contextNonInteractive` 命令自动激活（`supportsNonInteractive: true`）。可以通过：
```bash
claude --print "/context"
```
获取纯文本格式的上下文使用报告，包含各分区（system prompt/messages/tools）的 token 数和百分比。也可通过 SDK API 的 `get_context_usage` 控制请求获取结构化数据（与 `collectContextData` 共用数据层）。

**Q9：`/add-dir` 与直接修改 `CLAUDE.md` 中的目录配置有何区别？**

A：两者都能扩展 Claude 的工作目录访问权限，但有本质区别：
- `/add-dir` 是运行时操作，立即生效，无需重启；支持 `session`（临时）和 `localSettings`（持久）两种粒度
- 修改 `CLAUDE.md` 是静态配置，会话启动时加载，需要重启或 `/reload` 才生效
- `/add-dir` 通过 `applyPermissionUpdate` 操作 `toolPermissionContext`，属于权限控制体系；`CLAUDE.md` 是指令文档，由 AI 解析
- 沙箱模式下，`/add-dir` 会通过 `SandboxManager` 验证路径合法性，`CLAUDE.md` 修改则没有这层保障
