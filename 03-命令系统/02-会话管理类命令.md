# 会话管理类命令 — Claude Code 源码分析

> 模块路径：`src/commands/compact/`、`src/commands/clear/`、`src/commands/resume/`、`src/commands/rewind/`、`src/commands/session/`
> 核心职责：控制会话生命周期——压缩上下文、清除对话、恢复历史会话、回滚检查点
> 源码版本：v2.1.88

## 一、模块概述

会话管理类命令维护 Claude Code 的对话状态，是上下文窗口管理和会话连续性的核心。`/compact` 负责在不丢失语义的前提下压缩对话历史；`/clear` 重置当前会话；`/resume` 从历史记录中恢复会话；`/rewind`（别名 `/checkpoint`）通过消息选择器回滚到对话中的某个节点；`/session` 为远程模式显示 QR 码和 URL。

## 二、架构设计

### 2.1 核心类/接口/函数

**`compact`（`src/commands/compact/`）**
类型：`LocalCommand`，`supportsNonInteractive: true`。是 Claude Code 中逻辑最复杂的命令之一，包含三条压缩路径：Session Memory 压缩（轻量级，优先尝试）、Reactive 压缩（功能开关控制）、传统压缩（fallback）。通过 `microcompactMessages` 先做 token 预处理再压缩。

**`clear`（`src/commands/clear/`）**
类型：`LocalCommand`，`supportsNonInteractive: false`。核心是 `clearConversation()` 函数，执行 SessionEnd 钩子、发送缓存逐出提示、重置 sessionId、清除所有会话缓存、执行 SessionStart 钩子，完成一次完整的会话重置。

**`resume`（`src/commands/resume/`）**
类型：`LocalJSXCommand`，支持别名 `continue`，`argumentHint: '[conversation id or search term]'`。渲染会话列表 UI，支持模糊搜索，从 `~/.claude/projects/<hash>/` 目录读取 transcript 文件。

**`rewind`（`src/commands/rewind/`）**
类型：`LocalCommand`，别名 `checkpoint`。实现极简：调用 `context.openMessageSelector()` 打开消息选择器 UI，用户选择消息后会话回滚到该点。返回 `{ type: 'skip' }` 不追加任何消息。

**`session`（`src/commands/session/`）**
类型：`LocalJSXCommand`，仅在远程模式（`getIsRemoteMode()`）下启用，别名 `remote`。显示远程会话 URL 和 QR 码，供移动端扫码连接。

### 2.3 关键数据流

```
/compact [custom instructions]
    ↓
getMessagesAfterCompactBoundary(messages) → 过滤已压缩边界
    ↓
trySessionMemoryCompaction()   → 成功则直接返回（轻量）
    ↓（失败则继续）
isReactiveOnlyMode()?
  是 → compactViaReactive()   → reactiveCompactOnPromptTooLong()
  否 → microcompactMessages() → compactConversation() → 传统压缩
    ↓
返回 { type: 'compact', compactionResult, displayText }
    ↓
REPL 用 compactionResult.messages 替换会话历史
```

## 三、核心实现走读

### 3.1 关键流程

**`/compact` 三路压缩决策：**
1. 过滤 `getMessagesAfterCompactBoundary` — 去除 snip 标记前的消息（避免重复压缩）
2. 若无自定义指令，先尝试 Session Memory 压缩（`trySessionMemoryCompaction`）——依赖 session memory 服务，速度快，不调用 LLM
3. 若 Session Memory 不可用，检查 `isReactiveOnlyMode()`——Reactive 模式下走 `compactViaReactive`，并发执行 Pre-Compact 钩子和缓存参数构建
4. 默认 fallback：`microcompactMessages`（减少 token）+ `compactConversation`（LLM 摘要生成）
5. 成功后：清除 `getUserContext` 缓存、运行 `runPostCompactCleanup`、`markPostCompaction`、`suppressCompactWarning`

**`/clear` 会话重置流程：**
1. 执行 `executeSessionEndHooks('clear', ...)` — 有 `CLAUDE_CODE_SESSIONEND_HOOKS_TIMEOUT_MS` 超时限制（默认 1.5s）
2. 发送 `tengu_cache_eviction_hint` 分析事件（提示推理服务释放提示缓存）
3. 调用 `regenerateSessionId()` 生成新的会话 UUID
4. 清除会话级别所有缓存（`clearSessionCaches`）
5. 执行 `processSessionStartHooks` — 为新会话初始化钩子

### 3.2 重要源码片段

**`/compact` 三路决策逻辑（`src/commands/compact/compact.ts`）**
```typescript
// 路径1：Session Memory 压缩（轻量，无需 LLM）
if (!customInstructions) {
  const sessionMemoryResult = await trySessionMemoryCompaction(messages, context.agentId)
  if (sessionMemoryResult) { /* ... 直接返回 */ }
}
// 路径2：Reactive 模式（Feature Flag）
if (reactiveCompact?.isReactiveOnlyMode()) {
  return await compactViaReactive(messages, context, customInstructions, reactiveCompact)
}
// 路径3：传统压缩（microcompact + LLM 摘要）
const microcompactResult = await microcompactMessages(messages, context)
const result = await compactConversation(microcompactResult.messages, ...)
```

**Reactive 压缩的并发优化（`src/commands/compact/compact.ts`）**
```typescript
// 钩子执行与缓存参数构建相互独立，并发进行
const [hookResult, cacheSafeParams] = await Promise.all([
  executePreCompactHooks(
    { trigger: 'manual', customInstructions: customInstructions || null },
    context.abortController.signal,
  ),
  getCacheSharingParams(context, messages),
])
```

**`/clear` 的完整清理链（`src/commands/clear/conversation.ts`）**
```typescript
// SessionEnd 钩子有超时保护，防止阻塞清除操作
await executeSessionEndHooks('clear', {
  signal: AbortSignal.timeout(sessionEndTimeoutMs), // 默认 1.5s
  timeoutMs: sessionEndTimeoutMs,
})
// 重新生成会话 ID，彻底隔离新旧会话
regenerateSessionId()
await clearSessionCaches(...)
await processSessionStartHooks(...)
```

**`/rewind` 极简实现（`src/commands/rewind/rewind.ts`）**
```typescript
// 命令本身只打开选择器 UI，具体回滚逻辑在 UI 层
export async function call(_args, context) {
  if (context.openMessageSelector) {
    context.openMessageSelector()
  }
  return { type: 'skip' } // 不追加任何消息
}
```

### 3.3 设计模式分析

**责任链模式（Chain of Responsibility）**：`/compact` 按 SessionMemory → Reactive → 传统三条路径依次尝试，每条路径负责判断自身是否适用并处理，不适用则传递给下一条链节点。这使得新的压缩策略可以插入链中而不改变其他路径。

**模板方法模式（Template Method）**：`compactViaReactive` 和传统压缩路径共享相同的后处理步骤（`setLastSummarizedMessageId(undefined)`、`runPostCompactCleanup()`、`suppressCompactWarning()`、`getUserContext.cache.clear()`），但核心压缩逻辑不同。

**命令模式（Command Pattern）**：`/rewind` 是最纯粹的命令模式实现——命令对象（`rewind`）只负责触发操作（`openMessageSelector`），不包含任何状态或逻辑，执行逻辑完全在接收者（消息选择器 UI）中。

## 四、高频面试 Q&A

### 设计决策题

**Q1：`/compact` 为什么需要三条压缩路径，而非统一一套算法？**

A：三条路径代表不同的可用性和质量权衡。Session Memory 压缩是最新、最轻量的实现，直接利用持久化的会话摘要，不消耗额外 token；但它依赖于 Session Memory 服务可用且有有效摘要，且不支持自定义压缩指令。Reactive 压缩是专为"响应式"（按需触发）场景设计的路径，支持批量处理消息组，减少 LLM 调用次数。传统压缩是最古老、最保守的兜底方案，通过 `microcompact`（规则剔除冗余内容）+ LLM 摘要生成的两步走方式，兼容性最好。三者不互斥但有优先级，系统总能找到一条可用路径。

**Q2：`/clear` 中执行 SessionEnd 钩子为什么要设置 1.5s 超时？**

A：SessionEnd 钩子（`CLAUDE_CODE_SESSIONEND_HOOKS_TIMEOUT_MS`）是用户定义的外部脚本，可能含有不可控的延迟（网络请求、文件写入等）。如果不加超时，用户执行 `/clear` 后可能需要等待数秒才能看到清除完成。1.5s 是"足够大多数轻量钩子完成，同时不阻塞用户体验"的经验值。超时后钩子被强制终止（通过 `AbortSignal.timeout`），会话清除继续执行——这是可用性优先于钩子完整性的设计决策。

### 原理分析题

**Q3：`/compact` 后为什么需要清除 `getUserContext.cache`？**

A：`getUserContext()` 返回用户上下文（如内存文件、技能信息），这些信息被 memoize 缓存以避免重复读取。压缩后，系统提示词（system prompt）会重新构建，其中包含的上下文也需要刷新。若不清除缓存，新会话使用的系统提示词可能包含过时的用户上下文（如已更新的 `CLAUDE.md` 内容）。`runPostCompactCleanup()` 处理其他缓存，`getUserContext.cache.clear()` 专门处理用户上下文缓存——两者都在压缩成功的所有路径中执行。

**Q4：`getMessagesAfterCompactBoundary` 的作用是什么？**

A：Claude Code 的 REPL 保留所有消息（包括已被 snip 标记的旧消息）用于 UI 滚动回放，但这些 snip 前的消息不应再次被 LLM 摘要——它们已经是"死"历史。`getMessagesAfterCompactBoundary` 通过查找 snip 边界标记，只返回边界之后的"活"消息，确保压缩引擎只处理有效上下文，避免重复压缩已压缩内容。

**Q5：`/session` 命令为什么只在远程模式下可见？**

A：`/session` 的作用是生成当前终端会话的远程访问 URL 和 QR 码，供移动设备或其他终端接入。这个命令仅在 `--remote` 标志启动（`getIsRemoteMode() === true`）时有意义——非远程模式下，终端就是全部，不需要远程接入 URL。`isHidden: !getIsRemoteMode()` 和 `isEnabled: () => getIsRemoteMode()` 双重控制确保命令在命令补全列表中不出现（`isHidden`），且即使直接调用也会被拒绝（`isEnabled`）。

### 权衡与优化题

**Q6：`suppressCompactWarning()` 在压缩后调用的目的是什么？**

A：Claude Code 在上下文窗口使用率超过阈值时，会显示"Context left until auto-compact: X%"的警告。用户手动执行 `/compact` 后，上下文已经大幅缩减，立即再次显示警告会造成困惑（甚至在某些情况下警告会在压缩结果显示前短暂出现）。`suppressCompactWarning()` 设置一个短暂的抑制标志，防止这种视觉噪音。这是用户体验层面的细节优化，与压缩核心逻辑无关。

**Q7：`/compact` 和 `/clear` 在上下文管理上有何本质区别？**

A：`/compact` 是**有损压缩**——保留对话语义摘要，AI 仍记得之前讨论的内容，但细节可能丢失；它通过 LLM 生成摘要替换原始消息，消耗额外 token 但保持连续性。`/clear` 是**完全重置**——彻底删除所有对话历史，AI 从零开始，不保留任何上下文；不消耗额外 token，但代价是失去完整的历史记忆。选择哪个取决于场景：长时间开发会话中想"瘦身"用 compact；想完全切换话题或任务用 clear。

### 实战应用题

**Q8：如何利用 `/compact` 的自定义指令实现特定的摘要格式？**

A：`/compact [custom instructions]` 的参数直接传入 `compactConversation` 作为自定义摘要指令，例如：
- `/compact Keep focus on the current bug and ignore UI discussions` — 专注于 bug 修复上下文
- `/compact Summarize in Chinese, keep all code snippets` — 中文摘要但保留代码
- `/compact Create a structured TODO list from the conversation` — 生成待办清单格式
注意：自定义指令会绕过 Session Memory 压缩路径（该路径不支持自定义指令），直接进入 LLM 摘要路径。

**Q9：`/rewind` 与 `/branch` 在会话管理上有何配合使用的场景？**

A：典型场景：
1. AI 走错了方向，用 `/rewind` 选择分叉点，回滚到正确的消息节点
2. 想同时探索多条解决路径，先用 `/branch` 创建分叉会话，在一条路走不通后 `/rewind` 回到分叉点再尝试另一路
3. `/rewind` 回滚后继续对话产生新历史，与原历史形成自然分叉（无需显式调用 `/branch`）
关键区别：`/rewind` 在当前会话内回滚（截断消息列表），`/branch` 创建独立的新会话副本，两者可以结合使用构成非线性的对话图。
