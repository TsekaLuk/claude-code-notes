# 其他命令 — Claude Code 源码分析

> 模块路径：`src/commands/btw/`、`src/commands/plan/`、`src/commands/memory/`、`src/commands/help/`、`src/commands/mcp/`、`src/commands/skills/`、`src/commands/plugin/` 及更多
> 核心职责：提供辅助功能——侧边提问、计划模式、记忆管理、MCP 管理、插件系统等未归类命令
> 源码版本：v2.1.88

## 一、模块概述

本篇涵盖 Claude Code 命令系统中不属于前述专题分类的其余命令，按功能特性分为五小类：

1. **对话辅助**：`/btw`（侧边提问）
2. **工作流管理**：`/plan`（计划模式）、`/memory`（记忆管理）
3. **扩展生态**：`/mcp`（MCP 服务器管理）、`/skills`（技能管理）、`/plugin`（插件管理）、`/reload-plugins`（重载插件）
4. **会话操作**：`/help`（帮助）、`/exit`（退出）、`/rename`（重命名）、`/share`（分享，内部）
5. **缓存操作**：`break-cache`（内部缓存工具）

这些命令展示了 Claude Code 命令系统的广度，也体现了不同实现策略的多样性。

## 二、架构设计

### 2.1 核心类/接口/函数

**`btw`（`src/commands/btw/`）**
类型：`LocalJSXCommand`，`immediate: true`。发起"侧边问题"——在不打断主对话上下文的情况下，启动一个轻量的并行 AI 查询。核心是 `buildCacheSafeParams()` + `runSideQuestion()`，通过 `getLastCacheSafeParams()` 优先复用主线程最后一次请求的系统提示词（字节完全相同），实现**提示词缓存命中（prompt cache hit）**。使用次数通过 `saveGlobalConfig({ btwUseCount })` 追踪。

**`plan`（`src/commands/plan/`）**
类型：`LocalJSXCommand`，参数提示 `[open|<description>]`。切换或查看当前会话的计划模式。计划模式下 AI 先生成结构化执行计划，等待用户确认后再执行，避免大型操作的误操作。

**`memory`（`src/commands/memory/`）**
类型：`LocalJSXCommand`。提供 Claude 记忆文件的可视化编辑界面，支持查看、编辑和管理 `~/.claude/memory.md`（全局记忆）和项目级 `CLAUDE.md` 文件。

**`mcp`（`src/commands/mcp/`）**
类型：`LocalJSXCommand`。MCP（Model Context Protocol）服务器管理入口，支持查看已连接服务器、安装新服务器、查看工具列表和诊断连接问题。

**`skills`（`src/commands/skills/`）**
类型：`LocalJSXCommand`。展示所有可用技能（Skill）的列表，支持按来源（内置/插件/用户自定义）筛选，帮助用户了解 AI 可以调用的技能。

**`plugin`（`src/commands/plugin/`）**
类型：`LocalJSXCommand`。插件管理命令，支持安装、卸载和查看插件，是 Claude Code 插件生态的用户入口。

### 2.3 关键数据流

```
用户执行 /btw <question>
    ↓
检查 args.trim() — 无参数则返回使用说明
    ↓
saveGlobalConfig({ btwUseCount: n+1 })   # 追踪使用次数
    ↓
渲染 BtwSideQuestion 组件
    ↓
buildCacheSafeParams(context)
  → getLastCacheSafeParams()  → 复用主线程最后一次系统提示词（缓存命中优先）
  → 失败则 Promise.all([getSystemPrompt, getUserContext, getSystemContext])
    ↓
runSideQuestion({ question, cacheSafeParams })
  → 独立 API 调用（不进入主线程消息历史）
    ↓
展示响应（Markdown 渲染 + 滚动框）
按 Space/Enter/Escape 关闭，onDone({ display: 'skip' }) 不追加消息
```

## 三、核心实现走读

### 3.1 关键流程

**`/btw` 缓存感知的侧边查询：**

`/btw` 的核心挑战是：如何在不干扰主对话的情况下，让侧边查询也能利用同样的上下文（系统提示词、用户上下文）？

1. `buildCacheSafeParams` 优先从 `getLastCacheSafeParams()` 获取主线程最后一次请求的完整参数——这些参数包含字节完全相同的系统提示词前缀，可以触发 Anthropic API 的提示词缓存，节省 token 成本
2. `stripInProgressAssistantMessage` 移除正在生成中的 assistant 消息（`stop_reason === null`），避免将未完成的回复传给侧边问题的上下文
3. `runSideQuestion` 独立发起 API 调用，结果通过 React state 更新 UI，不写入主线程 `messages` 数组
4. 用户按任意退出键（Escape/Space/Enter/Ctrl+C/D）时，`onDone(undefined, { display: 'skip' })` 关闭 UI 且不追加任何消息

**`/reload-plugins` 的缓存清除链：**
1. 调用 `clearCommandsCache()` 清除所有命令相关缓存（技能、插件、工作流）
2. 缓存清除后，`getCommands()` 下次调用会重新扫描插件目录、加载新技能
3. REPL 层监听到命令列表变更，更新 typeahead 补全和帮助信息

### 3.2 重要源码片段

**`/btw` 缓存感知参数构建（`src/commands/btw/btw.tsx`）**
```typescript
async function buildCacheSafeParams(context): Promise<CacheSafeParams> {
  // 移除正在生成中的消息，避免传入未完成的 assistant 内容
  const forkContextMessages = getMessagesAfterCompactBoundary(
    stripInProgressAssistantMessage(context.messages)
  )
  // 优先复用主线程缓存参数（字节完全相同 → 提示词缓存命中）
  const saved = getLastCacheSafeParams()
  if (saved) {
    return { ...saved, toolUseContext: context, forkContextMessages }
  }
  // 回退：重新并发构建（可能 miss 缓存）
  const [rawSystemPrompt, userContext, systemContext] = await Promise.all([
    getSystemPrompt(...), getUserContext(), getSystemContext()
  ])
  return { systemPrompt: asSystemPrompt(rawSystemPrompt), ... }
}
```

**`/btw` 使用次数追踪（`src/commands/btw/btw.tsx`）**

export async function call(onDone, context, args) {
  const question = args?.trim()
  if (!question) {
    onDone('Usage: /btw <your question>', { display: 'system' })
    return null
  }
  // 追踪 btw 使用次数，可能用于功能分析

  return <BtwSideQuestion question={question} context={context} onDone={onDone} />
}

### 3.3 设计模式分析

**侧车模式（Sidecar Pattern）**：`/btw` 是"侧车"（sidecar）的完美实现——附着于主对话但独立运行，共享上下文（系统提示词、历史消息）但不污染主会话状态。侧车模式的关键是"共享基础设施，独立执行"，`buildCacheSafeParams` 共享缓存参数，`runSideQuestion` 独立 API 调用，`display: 'skip'` 确保不写入历史。

**提示词缓存优化（Prompt Cache Optimization）**：`/btw` 通过 `getLastCacheSafeParams()` 复用主线程的系统提示词，这不是偶然的优化，而是精心设计的——相同的系统提示词字节序列触发 Anthropic API 的 prompt caching，显著降低侧边问题的成本和延迟。注释明确说明回退路径（重新构建）"may miss the cache"。

**懒惰计数（Lazy Counting）**：`btwUseCount` 通过 `saveGlobalConfig` 持久化到全局配置文件，这是用户行为分析数据的收集方式之一。与服务端事件（`logEvent`）的匿名遥测不同，`btwUseCount` 是本地计数，可能用于触发"新手引导"或"功能发现"提示。

## 四、高频面试 Q&A

### 设计决策题

**Q1：`/btw` 为什么使用 `immediate: true`，且关闭时使用 `display: 'skip'`？**

A：`immediate: true` 让 `/btw` 在 AI 推理进行时也能立即打开——用户可能在等待 AI 完成一个长时间任务时，有一个快速的侧边问题（如"这个函数的时间复杂度是多少？"），无需等待主推理完成再提问。`display: 'skip'` 确保 `/btw` 的提问和 AI 回答不出现在主对话历史中——侧边问题的本意是"不打断主对话"，若追加到消息历史，会干扰主线任务的上下文（后续 AI 看到对话历史时会看到这些无关提问）。两者共同实现了"零痕迹侧边问答"的用户体验目标。

**Q2：为什么 `/btw` 要追踪 `btwUseCount`？这些数据如何使用？**

A：`btwUseCount` 是本地使用频率计数器，保存在 `globalConfig`（`~/.claude/global_config.json`）中。可能的用途：产品团队通过分析这个指标了解功能使用频率（判断是否值得继续投资）；用于新功能发现机制（如首次使用后显示提示，或使用 N 次后解锁高级提示）；可能上报到遥测系统（若用户同意）帮助优化功能设计。与 `logEvent` 服务端事件不同，`btwUseCount` 即使在网络离线时也持续累计，体现了"本地优先"的数据策略。

### 原理分析题

**Q3：`/plan` 的计划模式如何影响 AI 的执行方式？**

A：计划模式（plan mode）通过修改 `AppState.planMode` 标志，改变主推理循环的行为：启用后，AI 在开始执行任何操作（写文件、运行命令）之前，先输出结构化的执行计划（通常是编号步骤列表），并等待用户通过特殊 UI 确认。用户可以修改计划或跳过某些步骤。这本质上是在 AI 和用户之间引入了一个"审批节点"，适合高风险操作（大规模重构、删除文件等）。技术实现上，`planMode` 标志通过系统提示词注入（或工具调用前置检查）传递给模型，引导模型改变输出格式。

**Q4：`/memory` 编辑的记忆文件与系统提示词的关系是什么？**

A：`~/.claude/memory.md`（全局记忆）和项目级 `CLAUDE.md` 在每次对话开始时被读取，内容注入系统提示词的特定区段（`getUserContext()` 收集这些内容）。`/memory` 提供了一个方便的 UI 来编辑这些文件，而不必手动打开文件系统。重要的是：编辑后的内容在**当前会话中不会立即生效**（系统提示词已经构建完成）；需要执行 `/clear` 重置会话或等待下次启动，`getUserContext.cache.clear()` 才会读取新内容。`/compact` 后也会刷新（`compact` 流程会清除 `getUserContext.cache`）。

**Q5：`/mcp` 和 `/plugin` 在扩展性架构中扮演什么角色？**

A：Claude Code 的扩展架构分两层：MCP（Model Context Protocol）层和插件（Plugin）层。
- MCP 是外部协议标准，允许第三方服务器提供工具、资源和提示词，`/mcp` 管理这些外部服务器的连接和配置
- 插件是 Claude Code 特有的扩展格式，包含完整的命令、技能和钩子定义，`/plugin` 管理本地安装的插件包
两者共同构成 Claude Code 的"应用商店"，MCP 面向服务端集成（数据库、API、工具链），插件面向 CLI 功能扩展（新命令、新工作流）。`/reload-plugins` 在两层都触发刷新，`clearCommandsCache()` 清除所有已加载的插件和 MCP 命令缓存。

### 权衡与优化题

**Q6：`/btw` 的缓存命中策略有何局限性？**

A：`getLastCacheSafeParams()` 依赖于主线程在最后一次 API 调用后保存的参数（在 stop hooks 中捕获）。局限性：
1. **首次对话无缓存**：对话开始的第一轮，尚无 `saved` 参数，必须走回退路径重新构建
2. **特殊模式 miss 缓存**：若主线程使用了 `--agent`、`--system-prompt` 等特殊系统提示词扩展，回退路径重建的系统提示词可能与主线程不完全相同，导致缓存 miss
3. **并发请求冲突**：若用户在 AI 正在响应时立即调用 `/btw`，`lastCacheSafeParams` 可能是上一轮的，而非当前推理的参数
这些都是已知权衡，注释中已明确说明。在大多数场景下（非首轮、非特殊模式），缓存命中率很高。

**Q7：为什么 `/skills` 和 `/plugin` 需要独立命令，而不是通过 `/config` 的标签页访问？**

A：用户使用技能和插件的频率远高于修改配置设置。独立命令（`/skills`、`/plugin`）提供：
1. 快速直接访问，无需通过配置面板导航到对应标签页
2. 可以在 `CLAUDE.md` 中被 AI 系统引用（AI 可以建议用户执行 `/skills` 了解可用技能）
3. 可以被其他命令调用（如安装新插件后自动打开 `/skills` 展示新技能）
`/config` 面板适合偶尔需要调整的深度设置，而 `/skills` 和 `/plugin` 是高频度的功能发现和管理工具，入口路径越短越好。

### 实战应用题

**Q8：如何在复杂项目中高效利用 `/btw` 和 `/plan` 的组合？**

A：典型场景：
1. AI 正在执行大型重构任务时，用 `/btw "this approach looks risky, what are the alternatives?"` 在不中断任务的情况下咨询意见
2. 在启动复杂任务前用 `/plan` 让 AI 先展示执行计划，仔细审查后再确认执行
3. 计划执行中遇到疑问时用 `/btw` 快速查阅相关知识，不打断计划进度
4. `/plan` 确认后 AI 开始执行，用 `/btw` 实时跟踪某个子任务的进展
关键是 `/btw` 的"零痕迹"特性——它的提问不会影响 AI 对主任务的理解，可以频繁使用而不产生上下文污染。

**Q9：插件和 MCP 服务器的选择标准是什么？什么时候用插件，什么时候用 MCP？**

A：选择标准：
- **MCP 服务器**：适合需要访问外部服务或数据库的工具（Slack、GitHub、数据库查询）；需要实时数据；多个 AI 工具/应用之间共享（MCP 是开放协议）；需要跨平台复用
- **Claude Code 插件**：适合 Claude Code 专有功能扩展（新的斜杠命令、工作流脚本）；团队内部工作流标准化；与 Claude Code 特有功能深度集成（钩子、技能、命令系统）
简单判断：如果功能需要在多个 AI 工具中使用，选 MCP；如果功能只为 Claude Code CLI 设计，选插件。通过 `/mcp` 和 `/plugin` 可以动态管理两者，`/reload-plugins` 可热重载不需重启。
