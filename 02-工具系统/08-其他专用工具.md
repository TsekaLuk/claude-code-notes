# 其他专用工具 — Claude Code 源码分析

> 模块路径：`src/tools/SkillTool/`、`src/tools/ConfigTool/`、`src/tools/BriefTool/`、`src/tools/NotebookEditTool/`、`src/tools/TodoWriteTool/`、`src/tools/RemoteTriggerTool/`、`src/tools/SyntheticOutputTool/`、`src/tools/ScheduleCronTool/`、`src/tools/TeamCreateTool/`、`src/tools/TeamDeleteTool/`
> 核心职责：提供技能调用、配置管理、用户通信、Jupyter 编辑、任务管理、远程触发、结构化输出、定时任务和团队管理等专用功能
> 源码版本：v2.1.88

## 一、模块概述

其他专用工具是 Claude Code 工具系统中功能最为分散的一组，每个工具解决一个独立的垂直场景：

- **SkillTool** — 执行用户定义的技能（Skill），通过 fork 代理运行技能提示词
- **ConfigTool** — 读取/写入 Claude Code 配置项（主题、模型、权限等）
- **BriefTool** — 向用户发送带附件的消息（主动通知模式），用于后台任务完成通知
- **NotebookEditTool** — 编辑 Jupyter Notebook（`.ipynb`）文件，专门处理 JSON 格式的 notebook 结构
- **TodoWriteTool** — 会话级任务清单（全量替换模式，旧版 todo 系统）
- **RemoteTriggerTool** — 接收来自外部系统的远程触发事件
- **SyntheticOutputTool** — 在非交互模式（headless）下提供结构化输出格式
- **ScheduleCronTool（CronCreate/Delete/List）** — 管理定时任务（cron 任务创建/删除/列出）
- **TeamCreateTool / TeamDeleteTool** — 管理 Agent 蜂群（swarm）中的团队（Team）

## 二、架构设计

### 2.1 核心类/接口/函数

**`SkillTool`** — 技能执行工具

从 `getCommands()`（本地技能）和 `AppState.commands`（MCP 技能/提示词）中查找匹配的技能，通过 `prepareForkedCommandContext()` 准备 fork 上下文，调用 `runAgent()` 在独立代理上下文中运行技能。支持技能 frontmatter 中的 `model` 覆盖（`resolveSkillModelOverride()`）和递归技能调用（SkillTool 本身可在技能内被调用）。

**`ConfigTool`** — 配置读写工具

统一接口：`{ setting, value? }`——省略 `value` 时读取，提供 `value` 时写入。`supportedSettings.ts` 维护白名单配置项（`theme`、`model`、`permissions.defaultMode` 等），未列入白名单的键名被拒绝（防止 AI 意外修改敏感配置）。底层调用 `getGlobalConfig()`/`saveGlobalConfig()`。

**`BriefTool`** — 用户通知工具

接收 `{ message, attachments?, status }` 输入，`status: 'proactive'` 表示主动推送（用于任务完成、障碍通知等），`status: 'normal'` 表示回复用户。`attachments` 支持文件路径数组，通过 `resolveAttachments()` 上传并包含在消息中。仅在 Kairos 功能（`feature('KAIROS')`）激活时启用。

**`SyntheticOutputTool`** — 结构化输出工具

工具名 `StructuredOutput`，仅在非交互式会话（`isNonInteractiveSession: true`）下启用。使用 AJV（JSON Schema 验证库）动态验证输出内容满足调用者定义的 schema，确保机器可读的输出格式符合预期。输入模式为 `z.object({}).passthrough()`（允许任意字段）。

**`TeamCreateTool`** — 团队创建工具

调用 `writeTeamFileAsync()` 创建团队配置文件（`~/.claude/teams/{teamName}.json`），包含团队名、描述、成员列表、领队配置。调用 `registerTeamForSessionCleanup()` 注册会话清理钩子，确保会话结束时删除临时团队配置。`isEnabled()` 检查 `isAgentSwarmsEnabled()` — 仅在蜂群模式下可用。

### 2.3 关键数据流

**SkillTool 执行流程：**

```
输入: { skill: "my-skill", args: "参数文本" }
    ↓
getCommandsWithMcp(appState) → 合并本地+MCP技能列表
    ↓
findCommand(allCommands, "my-skill") → 找到技能定义
    ↓
parseFrontmatter(skillContent) → 提取 model/description 等元数据
    ↓
resolveSkillModelOverride(frontmatter.model) → 决定使用的模型
    ↓
prepareForkedCommandContext(command, args) → 构建 fork 上下文
    ↓
runAgent(forkedContext, ...) → 在独立代理中执行技能
    ↓
extractResultText(agentResult) → 提取最终结果文本
    ↓
输出: { result: "技能执行结果" }
```

**ConfigTool 读写逻辑：**

```
输入: { setting: "theme", value?: "dark" }
    ↓
isSupported(setting) → 检查白名单（theme/model/permissions.defaultMode 等）
    ├─ 不在白名单 → 拒绝（errorCode: 1）
    └─ 在白名单 → 继续
    ↓
value === undefined? → 读模式
    └─ getConfig(setting) → 返回当前值
value 提供? → 写模式
    ├─ getOptionsForSetting(setting) → 获取合法值列表
    ├─ 验证 value 是否在合法列表中
    └─ saveGlobalConfig({ [setting]: value }) → 持久化
    ↓
输出: { current: "dark", options: ["light", "dark", "auto"] }
```

## 三、核心实现走读

### 3.1 关键流程

**BriefTool 的主动通知设计：**

`status` 字段区分了两种使用场景：`'normal'` 是应答模式（用户问、AI 答），`'proactive'` 是主动推送模式（AI 主动汇报）。`proactive` 状态下，UI 层会使用不同的视觉处理（可能添加通知音/高亮），确保用户注意到这是来自 AI 的主动通知而非对话回复。后台任务完成时、遇到障碍时，AI 通过 `proactive` 状态的 BriefTool 通知用户，即使用户当时未在关注界面也能及时得到提醒。

**SyntheticOutputTool 的动态 schema 验证：**

工具在非交互模式下使用，调用者（API 客户端）通过系统提示词告诉 AI "你的最终输出必须通过 StructuredOutput 工具，使用如下 JSON Schema"。`SyntheticOutputTool` 在 `call()` 中使用 AJV 动态编译并验证输出内容，确保机器可读格式的结构正确性。这使 Claude Code 能作为微服务使用——调用者提供输出 schema，Claude Code 保证输出满足该 schema。

**TeamCreateTool 的蜂群配置写入：**

创建团队时，`writeTeamFileAsync()` 将团队配置写入 JSON 文件（路径 `~/.claude/teams/{teamName}.json`），包含：领队名称（`TEAM_LEAD_NAME`）、团队名称（`sanitizeName(team_name)`）、描述、成员配置（颜色分配通过 `assignTeammateColor()`）、模型配置。团队文件是蜂群模式中 Teammate 发现团队成员的核心数据结构，SendMessageTool 通过读取团队文件找到目标 Teammate 的 Agent ID。

### 3.3 设计模式分析

**命令模式（Command）**

SkillTool 是命令模式的实现：技能（Skill）是封装了特定任务逻辑的命令对象，SkillTool 是命令执行器（Invoker），通过技能名称查找并执行命令，调用者（模型）不需要知道技能内部实现。

**白名单策略（Allowlist Pattern）**

ConfigTool 使用显式白名单（`supportedSettings.ts`）而非黑名单保护配置项。黑名单容易出现"遗漏新增配置项"的安全漏洞，白名单则要求开发者主动声明每个 AI 可以访问的配置项，默认拒绝未知项，安全性更高。

**可观察性工厂（Observability）**

所有工具都通过 `logEvent()` 记录关键事件（SkillTool 记录技能使用、ConfigTool 记录配置变更、TeamCreateTool 记录团队操作）。这不是功能需求而是可观察性需求，帮助 Anthropic 了解功能使用模式，指导产品决策。事件名称以 `tengu_` 前缀命名（tengu 是代号），避免与其他系统事件混淆。

## 四、高频面试 Q&A

### 设计决策题

**Q1：SkillTool 为什么要用 fork 代理（`runAgent()`）而非直接执行技能提示词？**

A：技能（Skill）可能包含需要工具调用的复杂任务（如技能内部调用 BashTool、FileEditTool 等），不只是简单的文本生成。`runAgent()` 提供完整的工具调用循环，技能内可以有多轮 AI 推理 + 工具执行，技能作为一个独立的子代理完成复合任务后返回结果。同时，fork 代理有独立的上下文：技能的执行过程（中间步骤）不污染主会话的对话历史，主会话只看到技能的最终输出结果，保持会话整洁。

**Q2：BriefTool 的 `attachments` 字段为什么注释要求"必须保持 optional"？**

A：会话恢复（session resume）时会重放旧消息，包括历史 BriefTool 调用。如果旧消息中的 BriefTool 调用没有 `attachments` 字段（该版本功能未添加 attachments），但当前版本的 Zod schema 要求 `attachments` 为必填，则重放时 schema 验证会失败，导致会话无法正确恢复。保持 `attachments` 为 `optional` 确保了向前兼容——新旧消息格式都能通过 schema 验证，旧会话依然可以正常恢复。这是 API 版本兼容性设计的重要原则：可以添加可选字段，不可以将字段改为必填。

### 原理分析题

**Q3：SyntheticOutputTool 的 AJV 动态验证是如何工作的？**

A：AJV（Another JSON Validator）是一个 JSON Schema 编译器，可以将 JSON Schema 编译为高性能的验证函数。`SyntheticOutputTool` 在 `call()` 中接收调用者传入的 `output` 内容（通过 passthrough schema），同时从工具定义（由 main.tsx 创建时传入的 schema 参数）获取预期的 JSON Schema。AJV 编译该 schema，生成验证函数，对 `output` 进行验证。若验证失败，返回包含详细错误路径的 `TelemetrySafeError`，告知调用者哪个字段格式不符合要求。这使 Claude Code 能够作为有类型约束的 headless API 工具使用。

**Q4：TeamCreateTool 的颜色分配如何工作？**

A：`assignTeammateColor()` 从预定义的颜色池中为新 Teammate 分配颜色，颜色用于在 tmux 多窗格布局中区分不同的 Teammate。颜色分配策略是"尽量避免重复"——维护已使用颜色的集合，新成员从未用颜色中选择；若所有颜色都已使用，则循环使用。颜色信息写入团队文件（`~/.claude/teams/{teamName}.json`），Teammate 启动时读取自己的颜色配置，通过 tmux 的 `tmux set-option pane-border-status` 命令设置边框颜色，实现可视化区分。

**Q5：TodoWriteTool 中的 `verificationNudgeNeeded` 是什么机制？**

A：这是一个自动质量保障机制。当 `feature('VERIFICATION_AGENT')` 启用且 `tengu_hive_evidence` 特性标志为 true 时，TodoWriteTool 检测"关闭 3+ 项任务的关闭操作，且其中没有验证步骤"的情况。若检测到，在工具结果中注入提示："建议你在标记任务完成前添加一个验证步骤"。这个"结构性推拨"（structural nudge）不强制 AI 执行，但在 todo 列表关闭时机主动提醒 AI 验证工作质量，减少"完成但未测试"的情况。

### 权衡与优化题

**Q6：ConfigTool 的白名单设计对可扩展性有什么影响？**

A：白名单带来安全性，但增加了维护成本。每次 Claude Code 新增可配置项时，开发者需要主动将其添加到 `supportedSettings.ts` 白名单，否则 AI 无法通过 ConfigTool 访问该配置。这避免了"遗忘地将敏感配置暴露给 AI"的风险，但需要开发流程中有专门的步骤审查新配置项是否应该加入白名单。实践中，白名单应包含"对使用体验有直接影响但不影响安全性的配置"（如主题、模型），不包含"涉及认证、API 密钥或权限绕过"的配置。

**Q7：CronCreateTool（`ScheduleCronTool`）的定时任务是如何持久化的？**

A：定时任务配置存储在磁盘上（`utils/cron/` 管理），包含任务 ID、cron 表达式、要执行的提示词/命令、触发条件等。Claude Code 进程启动时加载已存储的 cron 配置并注册定时器。`CronDeleteTool` 按任务 ID 删除配置文件并取消对应定时器。`CronListTool` 列出所有当前活跃的定时任务。定时任务通常用于"每小时检查 PR 状态"、"每天汇报进度"等周期性自动化场景，是 Claude Code 作为持久化 AI 代理（而非一次性工具）能力的体现。

### 实战应用题

**Q8：如何使用 SkillTool 构建可复用的代码审查工作流？**

A：创建技能文件（如 `~/.claude/agents/code-review.md`）定义代码审查提示词：系统提示词描述审查标准和输出格式，必要时加 frontmatter 指定 `model: sonnet`（代码审查需要较强推理）。技能中可以引用其他工具（GrepTool 搜索问题模式、GlobTool 查找待审查文件等）。使用时，模型调用 `SkillTool({ skill: "code-review", args: "审查 src/auth/ 目录的安全性" })`，SkillTool 在独立代理中运行审查，返回结构化的审查报告。技能可以被不同项目、不同会话复用，形成组织级的标准化工作流。

**Q9：在自动化 CI/CD 场景中，SyntheticOutputTool 如何保证输出格式符合下游系统期望？**

A：在 headless 模式（`--output-format json` 等参数）下，调用者在启动 Claude Code 时通过系统提示词注入输出 schema 定义（JSON Schema 格式）。Claude Code 运行任务后，使用 `SyntheticOutputTool` 提交最终结果，AJV 验证结果是否满足 schema。若不满足（如缺少必填字段 `test_results`），工具返回详细错误，模型收到错误后重新生成符合 schema 的输出。这种方式确保了 CI/CD pipeline 的下游系统（如测试报告聚合器、部署决策器）能够可靠地解析 Claude Code 的输出，避免因输出格式不一致导致 pipeline 失败。
